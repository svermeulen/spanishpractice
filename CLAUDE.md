# Spanish Practice

Client-only web app for practicing Spanish through AI roleplay conversations. Fully static — there is no server. Each user supplies their own API keys (stored in their browser's localStorage) and the page calls the AI provider / ElevenLabs directly. Supports Anthropic, OpenAI, Google Gemini, and any OpenAI-compatible endpoint (OpenRouter, Groq, local servers, ...) — picked via the model dropdown. Deployed as a static site (GitHub Pages); also runnable locally as plain static files.

## Running

```sh
npm start            # serves public/ at http://localhost:3000 (python3 -m http.server)
```

No build step and no dependencies needed to run the app — `npm start` just serves the static `public/` folder. Open the app, click ⚙ Settings, pick a model, and paste the API key for that model's provider (Anthropic / OpenAI / Gemini, or a custom endpoint); audio works out of the box via the browser's built-in voices, and an optional ElevenLabs key upgrades to higher-quality TTS. Keys live only in the browser and are sent directly to the providers.

Deploy: pushing to `main` publishes `public/` to GitHub Pages via `.github/workflows/deploy.yml` (enable Pages once: Settings → Pages → Source: GitHub Actions). Served at the custom domain **spanishpractice.app** (the Pages `cname`, also asserted by `public/CNAME`); DNS is managed in Cloudflare (CNAME-flattened apex → `svermeulen.github.io`). The fallback Pages URL is `svermeulen.github.io/spanishpractice/`.

The app has **no dependencies and no build** — it's just static files in `public/`, servable by anything (`npm start` is a thin wrapper around `python3 -m http.server`; no `npm install` is ever needed). Scenarios are generated on the fly by the AI, so there's no Node tooling and no `scripts/` directory.

## Architecture

- `public/api.js` — the direct API layer (this is what used to be `server.js`). Multi-provider via a small adapter layer: the `PROVIDERS` registry (anthropic / openai / gemini / compatible) lists each provider's models with `$/Mtok` pricing, and every provider implements one `complete({ modelId, system, messages, schema, maxTokens })` → `{ text, usage }`. The neutral message shape is `[{role, content}]` with a string `system`; each adapter translates to its provider's request. `complete()` (provider-agnostic) plus `resolveModel` / `getModelOptions` / `hasKeyForModel` sit on top. The roleplay turns use JSON-schema structured output — Anthropic `output_config.format`, OpenAI `response_format` (strict), Gemini `responseSchema` (converted by `toGeminiSchema`: uppercased types, `additionalProperties` dropped, `propertyOrdering` set). All calls are direct browser→provider (Anthropic uses the `anthropic-dangerous-direct-browser-access: true` CORS opt-in; safe because it's the user's own key). Keys live in localStorage: `anthropicApiKey`, `openaiApiKey`, `geminiApiKey`, `elevenLabsApiKey`, plus `compatibleBaseUrl` / `compatibleApiKey` / `compatibleModel` for the custom endpoint (chosen via the `__custom__` dropdown sentinel; pricing unknown so its cost isn't tracked). Each result carries a `usage` payload (tokens + computed cost) shown as a session cost ticker. There is no default model — `model` starts as `""` (a disabled "Choose a model…" placeholder in the dropdown); `complete()` and `hasKeyForModel()` short-circuit on the empty value, so nothing runs until the user picks one. Pricing note: only the Anthropic numbers are authoritative; OpenAI/Gemini are approximate estimates — update them in `PROVIDERS` if they drift. Exposed functions, all called from `app.js`:
  - `apiChat({situation, history, message, model, strict, level})` — one roleplay turn. Uses structured outputs (`output_config.format`) to return JSON: `learner_translation`, `corrected_message`, `natural_message` (how a native would phrase it — empty unless meaningfully different from the correction), `notes`, `mistake_tags`, `reply_es`, `reply_en`. `strict` is the accent-checking setting; `level` is the CEFR difficulty.
  - `apiOpening({situation, model, level})` — the AI's in-character first message for a new conversation (just `reply_es` / `reply_en`, no correction fields). Called on start so the learner doesn't face a blank chat.
  - `apiTutor({history, question, transcript, model, level})` — freeform grammar Q&A side-thread; receives the conversation transcript as context.
  - `apiScenario({model, level})` — generates one fresh `{learner, ai}` roleplay scenario (structured output) using a random theme from `SCENARIO_THEMES` for variety, scaled to the level.
  - **Difficulty (CEFR A1–B2):** the `LEVELS` registry holds, per level, three prompt fragments — `reply` (how the AI partner speaks), `scenario` (how complex a generated situation is), `correction` (how strict/deep the feedback is). `getLevel(id)` resolves with a fallback to `DEFAULT_LEVEL` (`A1`); `getLevelOptions()` builds the `<select>`. The `level` id is threaded into `chatSystemPrompt(situation, strict, level)`, `scenarioSystemPrompt(level)`, and `tutorSystemPrompt(transcript, level)`, and the bare CEFR label is named in each prompt so the model leans on its own CEFR prior. Tune the calibration only in `LEVELS`.
  - `apiTts({text, sessionId, slow, gender})` + `ttsAvailable()` / `ttsHasElevenLabs()` — phrase-level audio. Two backends: **ElevenLabs** (`eleven_multilingual_v2`, premade voices) when a key is set, else the browser's built-in **speech synthesis** (free). `ttsAvailable()` is true if either exists (so 🔊 buttons show for everyone; `body.no-tts` only when neither). Voices are split into male/female pools (`TTS_VOICES`) and `ttsVoiceForSession(sessionId, gender)` picks a consistent one matching the scenario's `voice_gender`; the browser backend matches gender best-effort by voice name (`pickBrowserVoice`). ElevenLabs clips are cached in-memory per session (object URLs in `app.js`, keyed by gender+speed+text) — no disk cache.
- `public/` — vanilla JS frontend, no build step. `api.js` loads before `app.js`.
  - There is no start screen: once configured, each conversation begins with an **AI-generated scenario** (`apiScenario` in `api.js`, a cheap structured-output call on the chosen model with a random theme hint for variety), then opens in character. The 🎲 button (and `⌘K`) generates a fresh scenario; the ✎ button lets the learner type their own situation; the 🤖 button previews the hidden AI-facing prompt (on hover the header label flips from "Situation: …" to "AI Prompt: …" via `showingAiPrompt` / `renderSituationLabel`). Each scenario is a `{learner, ai}` pair: `learner` is the learner-facing description shown in the header (prefixed "Situation: "), `ai` is the AI partner's persona sent to the prompt (hidden from the UI). `normalizeScenario` in `app.js` also accepts a plain string (typed situations) by using it for both fields. Session setup is split into `resetSessionState()` (clears panes/state) + scenario application; `startRandomSession()` resets, shows a "Generating a situation…" placeholder, awaits `apiScenario`, then fires the opening.
  - 🔊 buttons (teacher replies, corrected sentence, "More natural" line) play TTS on demand; ⌥click plays slower. Starting a clip stops the previous one (`stopAudio` handles both the `Audio` element and `speechSynthesis`). With "Auto-play replies" on (default), each teacher reply auto-plays when audio is available — `playTts(..., auto=true)` fails silently if a browser blocks playback before the first user gesture.
  - The correction diff is computed client-side (`wordDiff` in `app.js`): word-level LCS aligned accent-insensitively (so `tu`→`tú` lines up as one fix, not a delete+insert). Real changes render as red strikethrough / green insertion. Accent/case/punctuation-only differences depend on the **"Correct accents & punctuation"** setting (`checkAccents`, off by default): lenient (default) renders them as a soft amber highlight and tells the AI not to flag/tag/note them (the learner's English keyboard has no accents); strict renders them as real strikethrough/insertion corrections and lets the AI tag/note them. The setting is passed to `apiChat`/`chatSystemPrompt` (prompt side) and `wordDiff`/`renderDiff` (diff side), and applies to subsequent messages only (past corrections aren't retroactively re-judged). `forgivenEqual` still forgives trailing `,;:.`/`¿¡`/sentence-start caps in lenient mode.
  - Voice input (`setupVoiceInput` in `app.js`, "Voice input" section): the 🎤 button in the chat input row dictates Spanish via the Web Speech API (`SpeechRecognition`/`webkitSpeechRecognition`, `lang: es-ES`, interim results shown live). The transcript fills the chat input (appended to any typed text) but is **not** auto-sent, so the learner can edit before pressing Send. Starting dictation calls `stopAudio()` so the mic doesn't transcribe TTS; submitting calls `stopVoiceInput()` so a late transcript can't refill a cleared input. Unsupported browsers (Firefox, older Safari) get `body.no-stt`, which hides the 🎤 button.
- Conversation state lives only in the browser — there is no server, and nothing is sent anywhere but the AI provider. The **active conversation** is mirrored to `localStorage` (key `activeSession`) so a reload resumes it; the "Session persistence" section in `app.js` handles it: `saveSession()` writes a `{v:1, sessionId, situation, situationDisplay, voiceGender, opening, turns, tutorTurns, cost, tokens}` snapshot after every completed turn (chat, tutor, and the opening), `restoreSession()` rebuilds the DOM + derived state (`chatHistory`/`transcript`) from it on boot (with `restoring=true` suppressing per-message autoplay), and `clearSavedSession()` runs whenever a **new** conversation starts (🎲 / typed / onboarding) — so only the in-progress conversation is kept, with no history of past chats and no mistake log. Audio clips aren't stored (regenerated on demand). Restore is best-effort: it only kicks in at boot when a usable model+key already exists, and a quota/parse failure silently falls back to a fresh scenario. Assistant turns are sent in history as `reply_es` only (not the full JSON).
- First run shows the **onboarding modal** (`#onboarding` in `index.html`, wired in the "Onboarding modal" section of `app.js`): it explains the app + bring-your-own-key/client-side model, collects a provider + key + a CEFR level (`#onbLevel`, defaulting to the stored `level`), picks that provider's cheapest model (`firstModelForProvider` — Haiku / gpt-4o-mini / Gemini Flash / the custom sentinel), writes everything (including `level`) to localStorage + reflects it into the Settings inputs (`syncSettingsInputs`, plus the model/level selects), and starts the conversation. Returning users (a usable model+key already stored) skip it. Closing the modal (×) drops into Settings as an escape hatch. The opening fires via `generateOpening()`, which self-guards on session+model+key+in-flight state (`openingInFlight`), so it runs exactly once and resumes automatically when config arrives (`maybeResumeOpening`).

## Keyboard shortcuts

Defined in the `KEYMAP` object in `public/app.js`; rebindable without code changes via localStorage key `keymap` (JSON, e.g. `{"toggleTranslations": "meta+t"}`).

- `Tab` — jump between the two text inputs
- `↑`/`↓` — shell-style recall of previously sent messages in either input (draft is preserved)
- `⌘K` — restart with a new random situation (same as the 🎲 button)
- `⌘E` — toggle auto-show translations (teacher EN + "Understood as"), `⌘I` — toggle auto-show notes; both flip the persisted settings (also exposed in the ⚙ settings popover)

## Settings popover

The ⚙ button in the header opens a popover with all per-user settings (persisted in localStorage, applied immediately): the model select (grouped by provider, built from `getModelOptions()`), the difficulty **level** select (`level`, CEFR A1–B2 from `getLevelOptions()`, default `A1`, applied to subsequent turns), per-provider API keys (Anthropic / OpenAI / Gemini, plus the optional ElevenLabs key that gates the 🔊 buttons), a collapsible "Custom OpenAI-compatible endpoint" section (base URL / key / model id), auto-show translations (`autoShowEn`), auto-show notes (`autoShowNotes`), correct accents & punctuation (`checkAccents`, default off / lenient — see the diff notes above), show session cost (`showCost`, default on), and auto-play replies (`autoPlayAudio`, default on — a `.tts-only` row hidden when no audio backend is available). Only the key/endpoint block for the selected model's provider is shown — `applyProviderVisibility()` toggles the `#set-anthropic` / `#set-openai` / `#set-gemini` / `#set-compatible` blocks on load and on every model change (the model select and ElevenLabs key always stay visible; the custom-endpoint disclosure auto-opens when its provider is active). Only the key for the selected model's provider is needed to chat; supplying it (or switching to a model whose provider is already configured) auto-resumes the opening. `Esc` or an outside click closes it.
